# Email Email Header Forensics — Implementation Explanation: Docker (Platform-Independent) Version

## 1. Why this version exists

The Vercel and Netlify implementations are functionally complete, but each depends on its platform's proprietary mechanism to attach HTTP security headers — `vercel.json` on one, `_headers` on the other. Three of the required protections (HSTS, `X-Frame-Options`, and CSP's `frame-ancestors`) only take effect as genuine HTTP response headers, so a bare "copy the files anywhere" deployment cannot satisfy every parameter of the requirement. The Docker version resolves this by shipping the web server *with* the application: an nginx instance with the entire header policy compiled into its configuration lives inside the image. Wherever the container runs — Docker Desktop on a Windows or macOS laptop, Docker Engine or Podman on Linux, a VPS, Kubernetes, a cloud container service, or an air-gapped host — the exact same server emits the exact same headers. The deployment target stops mattering, which is the definition of platform independence.

Nothing about the application changed. It is the same single-page React bundle in which all parsing, SPF/DKIM/DMARC evaluation, alignment checking, timestamp analysis, provider recognition, evidence correlation, scoring, interactive questioning, hop-map rendering, and report export execute inside the visitor's browser. The container is a delivery vehicle for static files; it receives no analysis data because the application contains no code path that could send any.

## 2. The image, stage by stage

The Dockerfile is a two-stage build. The first stage uses `node:22-alpine` purely as a disposable build environment: it installs the locked dependencies with `npm ci` (copying `package.json` and `package-lock.json` first, so Docker's layer cache skips reinstallation when only source changes), then runs `vite build` to produce the `dist/` output — one content-hashed JavaScript bundle of roughly 63 kB gzipped plus `index.html`. None of this tooling survives into the final image.

The second stage is the runtime, and its base image choice is a deliberate security decision: `nginxinc/nginx-unprivileged:1.27-alpine` rather than stock nginx. The unprivileged variant runs both the master and worker processes as a non-root user (uid 101) and listens on port 8080 instead of 80, which means the container needs no Linux capabilities at all and works under rootless Podman and restricted Kubernetes pod security policies without modification. Into this base go exactly two things: the compiled `dist/` directory and a hardened `nginx.conf`. The finished image is around 25 MB, contains no shell tooling beyond the Alpine base, no Node runtime, no package managers with network access, and defines a `HEALTHCHECK` that probes the site every thirty seconds so orchestrators can detect a wedged instance.

## 3. Where the headers live now

The nginx configuration is the platform-independent replacement for `vercel.json` and `_headers`. It declares the identical Content-Security-Policy — `connect-src` restricted to the site's own origin plus `dns.google` and `cloudflare-dns.com`, `frame-ancestors 'none'`, `form-action 'none'`, `object-src 'none'` — along with `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: no-referrer`, a restrictive `Permissions-Policy`, and cross-origin isolation headers, attached with the `always` flag so they appear even on error responses. The caching strategy is also carried over: hashed `/assets/*` files are served as `immutable` with a one-year lifetime, while `index.html` is `no-store`, so updates propagate on the next page load. Two privacy-specific choices go further than the managed platforms could: `access_log off` means the container does not record visitor IPs at all (Vercel and Netlify always keep edge logs), and `server_tokens off` hides the server version.

One header is intentionally absent from the container: HSTS. This is a correctness decision, not an omission. Browsers ignore Strict-Transport-Security over plain HTTP, and localhost use — the primary Docker Desktop scenario — is HTTP by design. Worse, a container blindly emitting HSTS behind an unknown reverse proxy could poison an entire hostname for a year. The setup guide therefore places HSTS where it belongs: at the TLS terminator, with a ready-made Caddy example that also provisions certificates automatically. When deployed publicly this way, the complete header set from the requirement is present, and on localhost the browser's secure-context treatment of `localhost` means nothing is actually degraded. As in the other two versions, the CSP is additionally embedded as a meta tag in `index.html`, so even a stripped or misconfigured server header leaves the egress allow-list enforced.

## 4. Runtime hardening

The included `docker-compose.yml` layers defense-in-depth on top of the already-unprivileged image. The root filesystem is mounted read-only with only a 16 MB tmpfs for nginx scratch space, so even code execution inside the container could not modify the served application. All Linux capabilities are dropped, `no-new-privileges` prevents any escalation, and memory and PID limits bound resource abuse. These settings cost nothing operationally because a static file server genuinely needs none of what they remove. The threat model this addresses is the same one the CSP addresses in the browser: assume a component might someday be compromised, and make the blast radius structurally small.

## 5. Distribution: how "anybody can pull and run"

The repository includes a GitHub Actions workflow that, on every push to `main` or version tag, builds the image for both `linux/amd64` and `linux/arm64` — covering Intel/AMD servers, Apple Silicon Macs running Docker Desktop, and ARM boards like Raspberry Pi — and publishes it to GitHub Container Registry under the repository's namespace. Once the package is marked public, the entire installation for any user on any platform is a single command: `docker run -p 8080:8080 ghcr.io/<owner>/header-forensics`. Users who prefer to audit before running (appropriate for a security tool) can instead clone and `docker build`, and the deterministic lockfile plus two-dependency surface (React and React-DOM only) keeps that audit tractable. For air-gapped environments, `docker save` and `docker load` move the image on removable media, after which the container runs with zero network access — only the optional browser-side DoH enrichment is affected, and the app degrades gracefully with the missing lookups flagged as uncertainties.

## 6. What is and isn't different from the Vercel/Netlify versions

Functionally nothing differs: same bundle, same analysis engine, same privacy model, same CSP, same caching semantics. The differences are operational. The Docker version gains guaranteed header delivery on any infrastructure, a no-visitor-logs posture, offline capability, and freedom from vendor configuration formats; it costs you the things managed platforms provide for free — TLS certificates, a global CDN, and DDoS absorption — which is why the public-exposure recipe pairs it with a TLS-terminating reverse proxy. The three versions coexist in the repository ecosystem without conflict, since each carries only its own configuration surface, and migrating between them never touches the application code.
